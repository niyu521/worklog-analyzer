## 0. 何を作るか(プロダクトの核・7/4夜 最終確定版)

### 一番大事なストーリー(ピッチの核)

**「Claude Codeで人が働く時代」が次に来て、その次に来るのは「その中身をどんどん自動化していく時代(もちろんhuman-in-the-loopは残る)」。この2つ目の時代を見越して、今のうちに必要なログを蓄積しておくためのサービス。** 審査員に一番刺さるのはここ。個々の技術の難易度ではなく、「今どのフェーズにいて、次に何が来るかを見据えている」という時間軸の先取りが差別化の核心。

### プロダクトの実体

**「Claude Code wiseな働く環境パッケージ」。** 既存サービス(WorkOS/Arcade.dev等の鍵管理、GBrain等のログ・分析基盤)をgatewayとして使いながら、社員がClaude Code経由で日常業務を行える環境そのものを提供する。この環境を使っているだけで：

1. 作業記録(ログ：手順・判断・例外処理)が副産物として自動的に溜まっていく
2. その記録は検索しやすい形に整理される
3. その記録を元に、「これスキル化しませんか?」(自走提案)、または「ここは完全自動で回せそうです、うちで/外部エンジニアリング会社に依頼して作りましょうか?」(受注・マーケットプレイス提案)が、ある程度自動で出てくる

**重要な設計判断**：スキル生成そのものをその場で自動生成・自動実行するところまでは製品の必須要件ではない。核心は「ログがうまく溜まり、整理され、的確な提案(自走 or 受注)が出てくること」。実際の自動化の実装(スキル生成)は、①顧客自身の自走、②実装マーケットプレイス経由での外部エンジニアリング会社への発注(**AIOTHはこのマーケットプレイスに参加する、身内であることを開示済みの実装パートナーの1社**であり、他の無関係な実装会社も同条件で参加する)、③パートナーのコンサル会社自身がこのツールで調査し自社実装する、という**複数の出口に分散させ、プロダクト企業自身が全案件を人手で捌く労働集約モデルにしない**(fable 5レビュー対応：「サービス業の皮を被ったツール」批判への回答。カスタム実装の労働はAIOTH分を含め全て社外に外出しし、透明に開示することがこの批判への直接の回答になる)。

**プラットフォームリスクへの対応(fable 5レビュー対応)**：ログ取り込み層はClaude Code専用ではなく、**Codex等の他のエージェントツールにも対応できる、エージェント横断の設計**にする。これにより「Anthropicが同種の管理者向け分析機能(実際に2025年8月にCompliance APIをローンチ済み)を自社で拡張したら終わり」というプラットフォームリスクを緩和し、特定ベンダーのエコシステムに閉じない中立的な作業ログ集約層というポジションを取る。企業が作業ログを外部に渡すことへの懸念(データ主権・プライバシー)は正面からエンジニアリングで担保する(テナントごとのデータ分離、暗号化、監査ログ、必要に応じたオンプレ/自社インフラ内完結のオプション)。

**技術構成(14:00確定)**：
- ログ取得は**二重チャネル**。①`CLAUDE.md`への書き込み指示(「何かやったらここに記録しなさい」という自然言語での自己申告。判断理由・例外処理はここで捉える)、②`PreToolUse`/`PostToolUse`フックによる構造化ログ(ツール名・タイムスタンプ等)
- **今回のデモ・提出物ではシングルテナント版のみを作る**(1社分のサンプルデータでログ取得→整理→提案までを完結させる)
- **将来展開(ピッチ末尾で一言触れる程度、実装はしない)**：プロダクト企業が中央サーバーをホストし、企業ごとにMCPエンドポイント・ログイン/テナント分離を発行するマルチテナントSaaSに拡張

**プロダクトの一文説明(7/4夜 最終改訂・決定版)**：「現場で実際どう作業が行われているかを把握するためのplugin」。

**重要：「今回SaaSとして立ち上げます」という提示の仕方はしない。** 今回作る・提出するものはあくまでシングルテナント版(1社分)。**ただし、見据えている将来像は明確にSaaSであり、その将来像を語ることがピッチの一部になる。**

**将来像(SaaS)のペルソナ構造**：
- **メインペルソナ＝AI化したい企業自身**
- **サブペルソナ＝コンサル会社**(クライアント企業に持ち込むツールとして使う。プロダクト企業自身は含まない——プロダクト企業は基盤の提供者であり、コンサル会社群はその利用者という立て付け)
- **プロダクト企業自身が、研修→FDE的にログ観察→提案、という動き方でこのサービスを使う**(ドッグフーディングと営業実演を兼ねる、サブペルソナ「コンサル会社」の実例)。**AIOTHはプロダクト企業とは別法人であり**、複雑な自動化案件を担う実装マーケットプレイスに参加する実装パートナーの1社に位置づけが変わる。身内の会社であることは明示的に開示するが、コンサル/FDEの中心的役回りはプロダクト企業側が担う

**チャネルコンフリクト・データ信頼性への対応方針(将来像を語る際に必須)**：サブペルソナとしてコンサル会社を語る以上、fable 5が指摘した懸念への対応はピッチの将来像パートで明確に言い切る。
- **個別案件は横取りしない**：他のコンサル会社・実装会社(AIOTHを含む)がこの基盤を使う場合、プロダクト企業は検出された個別の自動化案件を横取りしない。各実装パートナーがダッシュボードを見て、実装するかどうかを判断する。プロダクト企業はログ捕捉・検出のインフラ提供者に徹する(Stripeが加盟店の商売を奪わないのと同じ立ち位置)
- **横断活用は匿名集計のみ**：プロダクト企業が横断的に活用するのは、個別クライアントの具体的な業務ロジックではなく、匿名化・集計された全体トレンドのみ(「全体的にこの種の自動化ニーズが多い」という傾向を、自社の横展開サービス開発の判断材料にする程度)
- **AIOTHの位置づけの開示**：AIOTHはプロダクト企業と資本・人的関係のある身内の会社だが、実装マーケットプレイスに参加する数ある実装会社の1社にすぎない。無関係な外部実装会社も同条件で参加できる。この開示自体が「サービス業の皮を被ったツール」批判への正面からの回答になる——プロダクト企業本体の事業は検出・可視化・マーケットプレイス運営(正真正銘のプラットフォーム事業)であり、カスタム実装の労働(AIOTH分を含む)は全て社外に外出しされ、透明に開示されている
- **堀(モート)はテナントごとの粘着性が主**：「クロステナントでのノウハウ複利」ではなく、「その企業専用のBrainが使うほど賢くなり、乗り換えコストが上がる」というテナントごとの粘着性を主軸にする。集計トレンド活用は副次的な内部の製品開発材料であり、モートの主張とは別立てにする(fable 5指摘：この2つを同時にモートとして主張すると矛盾する)

## 1. どう使われるか(利用シーン)

1. **研修**でその会社にClaude/Coworkの使い方を導入する(プロダクト企業自身の顧客獲得の入口。ただしこの研修/コンサル自体はレッドオーシャンであると自覚した上で、その先に繋げるための入口と位置づける)
2. 社員は日常業務(返金対応・価格例外判断・インシデント対応のような、書かれない判断分岐を含む業務)を、Claude Code環境(gatewayパッケージ)経由・必要な部分はブラウザ拡張機能経由で行う
3. プロダクト企業はその会社に**FDE(Forward Deployed Engineer)的に入り込み**、数ヶ月ログを観察する。これは人間が現場に深く入り込まないと得られない「業務の解像度」を、AIエージェントの作業ログから自動的に・副産物として得る、という位置づけ
4. Brainが「この作業、繰り返しパターンです」とダッシュボードで提案(自走提案 or 受注・マーケットプレイス提案の2種類)
5. **簡単なものは顧客自身にスキル化してもらう**(自走を促す)、**複雑なものは実装マーケットプレイス経由で外部のエンジニアリング会社・フリーランスに案件として投げる**(AIOTHは、身内であることを開示済みの参加社の1社としてここに加わる。他の無関係な実装会社も同条件で参加する)
6. 自動化が増える→さらに使う→さらに溜まる→さらに賢くなる、のフライホイール
7. **さらにその先(ビジョン)**：多くの企業が今年〜来年に到達するであろう「Claude Code wise」な働き方の、さらに一歩先——「Claude API wise」(人間の介在がほぼ不要な状態)への移行を見据え、その移行に必要なログデータを今のうちから蓄積しておくためのツール、という立ち位置

## 2. どんなインパクトを起こすか

- **短期**：書かれない業務知識(SOPに残らない例外処理・判断ロジック)を可視化し、自動化候補として提示することで、属人化の解消と業務効率化を実現
- **中期**：プロダクト企業にとっては「研修→観察→自動化提案→マーケットプレイス外注(AIOTHはその実装パートナーの1社)」という新しい収益モデルの立ち上げ
- **長期(ビジョン、実装はしない)**：自動化が積み重なった先に、人間のPM/調整役がいなくても、VPS/中央ハブ上のmaster botがタスクの進捗を把握し、次の担当者に自動でアサインできる状態になる。最終的には**AIが会社を回し、人間には判断・対面が必要な仕事だけをAI側からアサインする**、という向きの逆転が起きる。これは1枚のロードマップスライドとして語るのみで、実装はしない

### ポジショニングの補足メモ(14:00追記、15:00競合調査により修正)

「AI導入コンサル/研修」というレッドオーシャンで戦うのではなく、**「多くの企業が来年には到達するであろう"Claude Code wise"な働き方の、さらに一歩先——"Claude API wise"(人間がほぼ介在しない状態)への移行ツールを、今のうちに作っておく」**という立て付けは、審査員にすぐ伝わる具体的な2段階モデルである。

これは今回の設計とも綺麗に繋がる：「Claude Code wise」の段階で作業ログを捕捉すること自体が、次の「Claude API wise」に飛ぶために必要なデータ収集そのもの、という説明ができるからである。研修/コンサルはあくまで入口(Red Ocean)であって、目的地はその先、という整理も明確になる。

**【15:00 競合調査による重要な修正】** この「2段階成熟度モデル」も「書かれたものvs実際にやっていること」という対比の言い回しも、**どちらも独自の発明として主張してはいけない**ことが競合調査で判明した。

- 成熟度モデル自体は、Gartner「Agentic AI Maturity Roadmap」、Microsoft「Agentic AI Adoption Maturity Model」(5段階)、OpenAI「Five Levels」(Chatbots→Reasoners→Agents→Innovators→Organizations)、SAE自動運転レベル準用の「エージェント自律性レベル0〜5」など、**既に複数の権威ある枠組みが存在**する。今回の2段階は、これらの多段階モデルの両端(低い段階と高い段階)だけを取り出して圧縮したもの、という位置づけになる
- 「書かれたものvs実際」という対比も、**Skan.ai**("observes how work actually gets done")、**Soroco Scout**("how work actually flows")、**Interloom**($16.5M調達、"70%の意思決定は文書化されない")、**Celonis**(プロセスマイニング業界の定番の謳い文句)が、既にほぼ同じ趣旨で使っている

**対応方針**：「独自のモデルを発明した」「独自の対比を発見した」とは主張しない。むしろ**Gartner/Microsoft/OpenAIといった権威ある枠組みを堂々と引用し、"このメガトレンドは本物である"という裏付けとして使う**。その上で、独自性の主張は**「Claude Code/Coworkという、生のAIエージェントセッションログを一次データソースにしている」という1点だけに絞り込む**(この角度で同じことをしている競合は、調査した範囲では見つかっていない)。

**【最終版】「Claude API wise」という自作用語は使わない。** 代わりに業界で既に確立された言葉を引用する：
- **「headless enterprise/agents」**(Salesforce・CIO.com等が使う、API層でエージェントが人間の介在なくオーケストレーションされる状態を指す用語)

**【7/4夜 修正】Dario Amodeiの「one-person billion-dollar company」は不採用。** これは新規に1人で創業する企業の話であり、狙っている未来(既存企業がAIで人員を大幅に減らしていく)とは方向が違う。代わりに**既存企業が実際にAIで人員削減を進めている実例**を引用する：
- **Amazon**：CEO Andy Jassyが「AIの効率化により全社的なコーポレート人員を削減していく」と明言。2026年1月に2度目の大規模レイオフ(1.6万人)を実施
- **Klarna**：CEO Sebastian Siemiatkowskiが、業務のAI自動化により2030年までに社員数を3,000人から2,000人に削減する方針を表明。既にAIアシスタント導入でカスタマーサポート約700人分を置き換え済み
- **Shopify**：CEO Tobi Lütkeが社内メモで「AIでできないことを証明できない限り増員は認めない」と明言

「我々が独自の未来を発明した」ではなく、**「Amazon/Klarna/Shopifyのような既存企業が既に向かっている"人員を大幅に減らしながら回る組織"に必要な、今のうちに集めておくべきデータ収集層」**という語り口に変える。

**競合の注意点**：FDEツール分野には**Nixo(YC S25)**「FDEのための初のオペレーションプラットフォーム」、**Hessian(YC P26)**「スケールできない業務を自動化する」という実在のYC出身直接競合がいる。ただしどちらも「AIエージェントのセッションログを一次データソースにする」角度では被っていない。なお一部のVC論考(a16z)は「FDEはソフトウェアで自動化できない」と逆の主張をしているが、**AWSが2026年6月30日に発表した10億ドル規模の"エージェントファースト"FDE組織**が、この方向性の実例として強い裏付けになる。

## 3. 採用が確定している設計判断

| 項目 | 内容 |
|---|---|
| ターゲット(ICP) | AIファーストで働くことを厭わない、変化に前向きな企業。レガシー企業は対象外 |
| ログ捕捉 | Claude Code/Coworkのセッションログ＋軽量なブラウザ拡張機能(MCPが届かないSaaS向け)。生ログは消失リスクがある前提で、構造化サマリー(1行要約＋手順・判断ポイント)を主役にする |
| ログ・分析基盤 | GBrain(Postgres/pgvector)を土台に流用。**ピッチでは正直に開示する**(主催者がGStack/GBrainの活用を推奨事項として明記しているため) |
| 鍵管理・権限 | WorkOS(AuthKit for MCP + auth.md)を第一候補、Arcade.devを次点。**これも正直に開示する**(「認証基盤は車輪の再発明せず、実績あるサービスを使う」という技術判断として) |
| 中央ハブの役割 | 「鍵の保管・払い出し」「ログの集約」の2つが必須機能。作業が実際に行われる場所(VPSか各自のPCか)はどちらでもよく、こだわらない |
| 権限の細かい制御 | ロールベースのアクセス制御＋時限付き例外申請という設計はあるが、**デモでは1枚のスライド説明に留め、作り込みはしない**(実ユーザーが1人のデモでは機能しない) |
| ビジネスモデル | 研修→利用→ログ蓄積→提案→(簡単:自走/複雑:実装マーケットプレイス外注。AIOTHは身内であることを開示済みの実装パートナーの1社)→フライホイール。トラクション主張は「自社を自社で回している」に限定(AIOTH全サービスは構想段階・顧客ゼロのため) |
| 差別化の立ち位置 | 「暗黙知プラットフォーム」としてGlean/Interloom/Cerenovusと直接対決しない。**「FDEが現場に入り込んで得る業務理解を、AIエージェントの作業ログから自動でスケールさせるツール」**という立ち位置。実際に競合調査で「①生のAIエージェントセッションログ②AIネイティブ企業がICP③ブラウザ拡張機能での橋渡し、の3つを同時に組み合わせている競合はない」という白地を確認済み。さらに「AI導入コンサル/研修(レッドオーシャン)」ではなく「"Claude Code wise"の次の段階"Claude API wise"(人間がほぼ介在しない状態)への移行ツール」という2段階成熟度モデルで語る(詳細は2節末尾の補足メモ参照) |
