# Company Brain — 技術リサーチまとめ

`2026-07-03-company-brain-architecture-design.md`(アーキテクチャ設計)の後に行った、個別トピックの技術リサーチの記録。設計書本体に反映する前の、生の調査結果集。

---

## 1. ターゲット顧客(ICP)の絞り込み

**結論**：ターゲットは「AIファーストで働くことを厭わない、変化に前向きなスタートアップ・企業」に絞る。レガシーでガチガチの企業(独自の古い社内アプリに依存し、変わる気がない企業)は今回のターゲット外とする。

**理由・効果**：
- fable 5が最初のコンセプトレビューで指摘した「全員がClaude経由で働く」という前提の**アダプション循環論法**が大幅に弱まる。既にAI経由で働きたい会社だけを相手にするなら、この前提はほぼ自明になる
- MCP/APIで完全に触れない独自の古い社内アプリの問題も、このターゲット層では深刻度が下がる。無理にレガシー企業まで対応しようとする「独自ブラウザ+PC監視ソフト」のような重量級の実装は不要になる
- 「対応SaaS/MCPはこれからも自然に増えていく」というトレンドに乗る設計でよい。今使えるものから素直に対応すればよく、無理な全網羅を目指さない。これは「human-in-the-loopをうまく回すためのワークスペース」であり、時間が経つほど人間の役割は自然に減っていく、というビジョンとも一致する

## 2. 作業ログの捕捉方法

### 2.1 Claude Code / Cowork

- Claude Codeのローカルセッションは`~/.claude/projects/<project>/<session-id>.jsonl`に自動保存される(公式ドキュメント確認済み)
- Claude Cowork公式説明：「Claude Code power for knowledge work」＝同エンジンのデスクトップ版自律作業ツール。セッション履歴はデバイス上にローカル保存、管理者がON/OFF切り替え可能
- **注意点**：Anthropic公式ヘルプに「プロジェクトをまたいだメモリは保持されない」旨の明記あり。実際に「セッション履歴・会話ログが消える」というmacOSでのバグ報告も存在([GitHub Issue #45076](https://github.com/anthropics/claude-code/issues/45076))
- → **生ログ(transcript)はいつか失われる可能性がある前提で設計する。** 構造化サマリー層を主役にし、生ログは「あれば見られるボーナス情報」として扱う

### 2.2 MCPが届かないSaaS(freee, 独自社内アプリ等)への対応

当初検討した「独自ブラウザ＋PC監視ソフトを1から作る」案(大阪AI Expoで見た製品からの着想)は、**実装規模が跳ね上がる**(独自ブラウザ・PC監視エンジン・VPN接続をフルスクラッチで作る規模)のと、**「社員を監視するソフト」という響きが審査員(特にガバナンス重視のSean Grove)への印象で諸刃の剣になる**リスクから不採用。

→ **着地点：ブラウザ拡張機能(Chrome extension等)に規模を縮小。** 独自ブラウザを作らず、拡張機能でページ遷移・操作の要約だけを拾う。実装規模が一桁小さくなり、対応SaaSも柔軟に後から追加できる。さらに、この拡張機能のUI自体を「役割ごとに使う機能が違う→検出パターンに応じてUIが個人ごとに最適化されていく」形にすれば、YC RFSテーマ②(Dynamic Software Interfaces)をCompany Brainの中の一機能として自然に接ぎ木できる(個人に繋がる種がない、という以前のテーマ②単体でのボツ理由は、Company Brainの一機能として出すことで解消される)。

## 3. GBrain(ログ格納・分析基盤)の調査

**結論**：GBrain(Garry Tan作、`github.com/garrytan/gbrain`)のPostgres/pgvectorエンジンを土台として流用する。

- 実体：`~/.gbrain/brain.pglite`にPGLite(組み込みPostgres)として存在。TypeScript/Bun製。「Postgres-native personal knowledge brain with hybrid RAG search」
- ベクトル検索：Voyage AI(`voyage-code-3`/`voyage-4-large`)またはOpenAI embeddingでpgvectorベースのハイブリッド検索(意味検索＋全文検索、RRFランキング)
- **`IngestionEvent`という汎用の取り込み契約**が既にあり、`IngestionSource`プラグインを1つ書くだけで新しいデータソースを追加できる。既存実装例(`gstack-learnings`)をテンプレートに改修すればよい
- **`subagent_tool_executions`、`context_volunteer_events`(session_id保持)、`ingest_log`、`timeline_entries`、`find_anomalies`**など、まさにセッション・イベント・ツール呼び出しを扱う前提のテーブルが既に存在
- GBrain自身の開発目標(北極星)が「company brainのための次世代Postgres」と明記されており、設計思想が一致
- **足りない部分(自作が必要)**：①パターン分析用のスキーマ拡張(`sessions`テーブル追加)、②VPS上での稼働形態の確定(Supabaseモード等)、③Cowork/Claude Code用の`IngestionSource`実装
- **リスク**：現状この端末のGBrainは古いロックファイルで接続不能・MCP未登録という、動いていない状態。着手前に`gbrain doctor`のリメディエーションフローで復旧させる必要あり
- 方針：GBrainは看板には出さず、裏側の技術選定として「必要な部分だけ食いちぎって使う」

## 4. 中央集権ハブの形：VPSは必須か？

### 4.1 発想の転換

当初「VPS＝作業する場所そのもの(SSHログインしてそこで作業する)」という前提で設計していたが、これを見直し。**中央ハブの必須の役割は3つだけ**と再定義：

1. **鍵の保管庫**：各種SaaS・クラウドDBへのMCP/API鍵を保管
2. **鍵の払い出し(権限管理)**：ロールに応じてスコープ付き・時限付きの鍵を、各人のAIエージェントに渡す
3. **ログの集約**：作業がどこで行われようと、ログは中央に集まる

**作業が実際に行われる場所(VPS上か、各自のPC上か)は問わない。** どちらが技術的にきれいなら、そちらを採用するというフラットな立場。

### 4.2 鍵ブローカー/MCPゲートウェイの選択肢比較

2026年時点で「MCPゲートウェイ」(スコープ付き・per-userの鍵管理をMCP経由で行う仕組み)という分野が急速に確立されつつある。2026年3月のMCP仕様改定でOAuth 2.1ベースの認可フローが標準化され、2025-11-25版でClient ID Metadata Document(CIMD)やStep-up認可(403 insufficient_scope)なども追加。ただし**MCP仕様自体は「1つのMCPサーバーが複数エンドユーザー分のアップストリームトークンをどう管理するか」までは規定しておらず、これは各実装/製品側の仕事**という点が確認できた。

#### 比較表

| 選択肢 | 成熟度・資金 | 強み | 弱み・リスク |
|---|---|---|---|
| **WorkOS**(AuthKit for MCP + auth.md) | 評価額$2B、ARR$30M、**顧客にOpenAI/Anthropic/Cursor**。2026年5月に agent向け認証プロトコル「auth.md」を公開 | 圧倒的に成熟・信頼性が高い。Anthropicが顧客という説得力 | MCP専用部分の価格表は一般のAuthKit料金に統合されており、専用の価格明記なし。「Pipes」機能はデモ段階の可能性 |
| **Arcade.dev** | 累計$72M調達(2026年6月シリーズA、Morgan Stanley/Wiproが戦略投資)。Fortune 500・大手米銀行での実運用実績 | 「権限が必要になったその場で要求する」という思想が今回の設計に一番近い。2025年11月にAnthropicと共同でMCP仕様に認証フロー(URL Elicitation)を採用させた実績 | 時限付き(time-boxed)の権限失効機能は公式ドキュメント上に見当たらず、自前実装が必要 |
| **Composio** | 累計$29M調達、SOC2/ISO27001取得、**顧客にGlean**、200社以上の実導入 | 統合数(2万以上のツール)が圧倒的 | スコープ管理はあるが時限付き失効は同様に未確認。認証の中身がブラックボックス寄り |
| **Klavis AI**(YC X25) | 資金$50万のみ、チーム3〜6人 | オープンソース寄り、100+統合 | 明らかに未成熟。スコープ/時限管理の裏付けなし |
| **IBM `mcp-context-forge`(OSS, セルフホスト)** | GitHub 4,000スター超、本日(7/3)も更新される活発なメンテナンス | **無料・per-userのOAuthトークンをアップストリームに転送する機能を公式に持つ**。VPS上に自前でホスト可能 | セルフホストの運用負荷は自分たちで持つ(ただし枯れたOSSなのでゼロから作るより大幅に楽) |
| **Descope**(Outbound Apps) | - | per-userのトークンボールト機能あり、Descope Outbound Appsが今回の要件にかなり近い | 詳細な資金・実績情報は未確認 |
| **Stytch**(Connected Apps) | - | MCP専用の認証機能はある | アップストリームSaaSへのper-userトークンブローカー機能は確認できず、クライアント⇄サーバー間のOAuthが中心 |

**AWS Secrets Manager+IAM、HashiCorp Vault(自前/HCP)、Doppler、1Password**は、いずれもサードパーティOAuthのper-user動的発行という今回の要件には不向き(Vaultは運用負荷・コストが重すぎる、Doppler/1Passwordの動的発行機能はベータ段階、AWSはAWS内リソース向けで外部SaaS OAuthとは相性が悪い)。ただし**Auth0 Token Vault**は例外で、サードパーティOAuthトークンの保管＋短命スコープ付きトークン発行という要件にピンポイントで合致する成熟プロダクト。

### 4.3 結論・推奨

**まずWorkOS(AuthKit for MCP + auth.md)でサクッと立ち上げてみて、詰まったらIBMの`mcp-context-forge`(OSS・セルフホスト)にフォールバックする。** Arcade.devはエージェント色を前面に出したい場合の次点候補、Auth0 Token Vaultは信頼性最優先の代替案として残す。

**注意**：どのプラットフォームを選んでも、「時限付き・単発閲覧で自動失効」という細かい例外アクセス申請ロジック自体は、自前で実装する前提で計画する(標準機能としては存在しないため)。
