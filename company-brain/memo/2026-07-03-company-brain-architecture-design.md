# Company Brain — アーキテクチャ設計

`2026-07-03-company-brain-final-concept.md`(コンセプト確定版)を受けて、システムアーキテクチャを詳細化したもの。この後fable 5による敵対的レビューにかける前段のドラフト。

---

## 0. 全体像

**今回作り込む部分(実働)**：VPS中央集権型で、ログ捕捉→Brain解析→自動化候補提示→承認→スキル化、のループを実装する。

**ピッチで語るビジョン(実装しない)**：ループが積み上がった先にある「AIが会社を回し、人間には判断・対面が必要な仕事だけをAI側からアサインする」PMレス状態。ロードマップの終着点として1枚のスライドで見せる。今回の実装は、そこに至るための「社内アーキテクチャの第一歩」という位置づけ。

---

## 1. VPS中央集権レイヤー

- VPSが全ての作業のハブになる。Notion、Gmail、スプレッドシート、MoneyForward、議事録ツールなどの外部サービスとMCP/APIで接続する。
- 各社員(user1, user2, user3...)はVPSにSSHログインし、Claude/Cowork経由で日常業務を行う。作業はVPS内で完結する(手元のPCにデータを同期して作業するわけではない)。

---

## 2. ログ・データ・権限レイヤー

### 2.1 データストア

自前でDBを設計するのではなく、**GBrain(Garry Tan作、Postgres/pgvectorベースの永続知識ベース)のエンジンを土台として流用する**。

理由：
- 既にPostgres(PGLite/Supabase切り替え可)＋pgvectorによるハイブリッド検索(意味検索＋全文検索)が実装済みで、ゼロから作るより大幅に工数が減る
- `IngestionEvent`という汎用の取り込み契約(`source_id`/`source_kind`/`timestamp`/`content`/`content_hash`/`metadata`)が既に存在し、プラグイン(`IngestionSource`)を1つ書くだけで新しいデータソースを追加できる
- `subagent_tool_executions`(誰が・いつ・どのセッションで・どのツールを使ったか)、`context_volunteer_events`(session_id保持)、`ingest_log`など、まさにセッション・イベント・ツール呼び出しを扱う前提のテーブルが既に存在する
- GBrain自身の開発目標(北極星)が「company brainのための次世代Postgre」と明記されており、設計思想が一致している

**足りない部分(自作が必要)**：
1. パターン分析用のスキーマ拡張(既存は文書中心のpages/chunksモデルのため、イベント・時系列分析向けに`sessions`テーブルを追加)
2. VPS上での稼働形態の確定(ローカルPGLiteではなくVPS集約型に。Supabaseモード or VPS上でPGLiteをサーバー的に稼働させる形を選定)
3. Cowork/Claudeセッションイベントを実際に流し込む`IngestionSource`の実装(既存の`gstack-learnings`ソースをテンプレートに改修)

方針：**GBrainは「必要な部分だけ食いちぎって使う」**。看板には出さず、裏側の技術選定として使う(以前の合意通り)。

### 2.2 ログの粒度：サマリー層 + 生ログ参照層の2段構造

Claude Code(ローカル)にもCowork(VPS上)にも「チャット/セッションごとのID」という単位が既にある。これを活かし、**全イベントを逐一構造化して溜め込むのではなく、2段構造にする**。

**`sessions`テーブル(GBrainのスキーマに追加)**

| フィールド | 内容 |
|---|---|
| `session_id` | セッションの一意ID(Claude Code/Coworkが元から発行するIDを流用) |
| `user_id` / `role` | 誰が・どのロールで |
| `task_context` | どの業務文脈か(タグ) |
| `started_at` / `ended_at` | セッションの開始・終了時刻 |
| `one_line_summary` | そのセッションで何をしたかの1行要約(安価なモデルで自動生成。GBrainの「Haikuで処理価値を判定する」既存ロジックと同じ発想) |
| `transcript_ref` | 生ログ(元のJSONLファイル等)へのパス・URI。**存在すれば見られるボーナス情報**として扱う |

**探索の流れ**：
1. Brainは基本的に`one_line_summary`の並びだけを見て「これ繰り返しパターンっぽい」と目星をつける(軽量・高速)
2. 確証が欲しい時だけ`transcript_ref`を辿って生ログを開き、詳細な手順を確認する(重い処理は必要な時だけ)

**注意点**：Claude Codeのローカルセッションは`~/.claude/projects/<project>/<session-id>.jsonl`に自動保存される。Cowork(Claude Codeと同エンジン)もローカル(=VPS)にセッション履歴を保存する仕様だが、Anthropic公式ヘルプに「プロジェクトをまたいだメモリは保持されない」旨の明記があり、実際に「セッション履歴が消える」バグ報告(GitHub Issue #45076)も存在する。**`transcript_ref`が指す生ログはいつか失われる可能性がある前提で設計し、`one_line_summary`だけでパターン検出が成立するようにしておく。**

### 2.3 権限モデル

- 正本データ・ログはGBrain(root配下のDB)に一元集約。
- 各ユーザーがSSH/Claude経由でアクセスする際、ロールに応じて見える/触れる範囲をDBの権限(Row Level Security的な仕組み)で絞る。ファイルをrsyncで複製する方式は同時書き込み衝突のリスクがあるため不採用とし、「同じ正本データに対する権限付きアクセス」に統一する。
- GBrainが元から持つ「read-write / read-only / deny」の権限ポリシーの考え方を参考に、ユーザー単位でこれを設定する。

**例外アクセス申請機能**：
- 通常はロール範囲内のデータしか見れない
- イレギュラーに権限外の資料が必要になった場合、VPS上でその場で承認者(上司)に「権限申請」を送信(Gmail/Slack等のMCP経由で通知)
- 承認されると、時限付き・単発の例外許可レコードがDBに追加され、一時的にアクセスを許可。一定時間経過 or 1回の閲覧後に自動失効
- 例外申請が頻発するパターンはBrainの検出対象にもなり、「正式にロールのルールを変えるべきでは」という二次提案に繋がる(将来拡張)

## 3. Brain解析レイヤー

- VPS内に溜まった`sessions`テーブルのサマリーを、専用subagent(Brain)が定期的に読む
- 同じ`task_context`内で、複数`session_id`にわたって手順の並びがどれだけ一致するかを見る、比較的シンプルな集計クエリでパターン検出を行う(AIモデルを都度呼ぶ必要はない)
- 「繰り返しパターン」「属人化」「陳腐化した手順」を検出し、ダッシュボードに自動化候補として提示する

---

## 4. 承認・実行レイヤー(ビジネスモデルと統合)

Brainが検出するのは「ここ自動化できそうです」という**提案止まり**。実際の自動化実装は2段階に振り分ける：

- **簡単なケース**：ダッシュボード上で「これ自動化・スキル化できます」と提示し、**顧客自身にスキル化してもらう**(自走を促す)
- **複雑なケース**：AIOTHが**コンサルティングとして入り込み、がっつり実装して納品する**(ここがエンジニアリング収益源)

ビジネスモデル全体(フライホイール)としては：
1. 入口＝研修でClaude/Coworkの使い方を教える
2. 研修を受けた社員が日常業務をClaude経由でやり始める
3. 作業ログが副産物として自動蓄積(セクション2の仕組み)
4. Brainが「ここ自動化できます」と提案
5. 簡単なものは自走、複雑なものはコンサル実装で収益化
6. 自動化が増える→さらに使う→さらに溜まる→さらに賢くなる、の自己強化ループ

トラクション主張は「AIOTHは顧客に売っている」ではなく「自分の会社を自分で回している」に限定する(AIOTH全サービスは構想段階・顧客ゼロのため)。

## 6. アーキテクチャ全体の依存関係まとめ

```
[外部サービス: Notion/Gmail/スプシ/MoneyForward/議事録]
        ↕ (MCP/API)
[VPS] ← SSHログイン ← [user1/user2/user3...(Claude/Cowork経由で作業)]
  │
  ├─ GBrain(Postgres/pgvector) ─ sessionsテーブル(サマリー+生ログ参照)
  │     ├─ 権限層(role-based access + 時限例外申請)
  │     └─ IngestionSource(Cowork/Claude Codeセッション取り込み)
  │
  ├─ Brain解析(定期実行の集計クエリ) → 自動化候補をダッシュボードに提示
  │
  └─ 承認 → 簡単:顧客が自走してスキル化 / 複雑:AIOTHがコンサル実装
              → MCP配線済みスキルとして定着 → 次回から自動実行 → ①に戻る
```
