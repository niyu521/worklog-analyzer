YC の RFS ページを確認しました。3テーマそれぞれについて、**YCが本当は何を求めているのか → なぜ今なのか → 5時間で作れて、かつ「会社の種」に見えるプロダクト案**の順で詳しく解説します。最後に「このハッカソンならどれを選ぶべきか」もまとめます。

## 前提：審査員から逆算した「勝ち筋」

先に結論の枠だけ。今回の審査員は偶然ではなく、テーマと直結しています。

- **Sean Grove（Linzumi）** … AIコーディングエージェントを「統制・レビュー・検証」する基盤を作っている人 → **エージェントの制御・信頼性**に反応する
- **Henry Ndubuaku（Cactus）** … スマホ/ウェアラブルでのオフラインAI推論 → **エンジニアリングの重さ・オンデバイス**に反応する
- **Garry Tan（YC CEO）** … GStack/GBrain の作者 → **「AIが自律的に動くための土台」**という思想そのもの

なので3テーマ共通で、審査員が刺さるのは「デモが動く」だけでなく **「AIエージェントが主役の世界を、地に足つけて設計できているか」**。ここを外さないのが大事です。

## テーマ1. Company Brain（Tom Blomfield）

### YCが本当に言っていること
ページの原文を要約すると、こうです。

> 会社の重要な知識は「人の頭の中」と、古いメール・Slackスレッド・サポートチケット・DBといった**バラバラの断片**に散らばっている。これをAIが使えないと、AIエージェントはまともに仕事ができない。

そこでYCが欲しいのは、単なる「ドキュメント検索」や「社内チャットボット」ではなく——

> **散らばった知識を引き出し、構造化し、常に最新に保ち、AI向けの "実行可能なスキルファイル（executable skills file）" に変換するシステム。**
> 「返金はどう処理するか」「価格の例外はどう決めるか」「エンジニアは障害にどう対応するか」といった、**会社の動き方そのものの "生きた地図"**。

キーワードは **「executable（実行可能）」**。ここが普通のRAG社内wikiとの決定的な違いです。Q&Aを返すだけじゃダメで、**その知識をエージェントが実際にアクションとして実行できる手順に落とす**ところまでがCompany Brain。Blomfield は「世界中の全ての会社がこれを必要とする」と言い切っています。

### なぜ今か
- LLMは賢くなったが「その会社固有の暗黙知」を知らない → ここが最後のボトルネック
- MCPの普及で「知識を各SaaSから吸い出す配管」が現実的になった
- まさに GBrain がこの縮図。今回セットアップした GBrain の発想を「会社レベル」に拡張するイメージ

### プロダクト案（5時間で作れる粒度）

**案A：「暗黙知 → 実行手順」への変換エンジン（本命）**
- Slack/メール/チケットのエクスポートを食わせる → LLMが「この会社ではこういう時こうする」という**Playbook（手順書）を自動抽出**
- ただの要約で終わらせず、各Playbookを **MCPツール呼び出し付きの実行フロー**として出力（例：「返金対応」→ Stripe APIで返金を叩くステップまで含む）
- デモ：「顧客が返金を要求してきた」と入力 → Brainが過去の判断を根拠に手順を提示し、承認したら実際にアクション実行
- **勝ちポイント**：検索ではなく**実行**まで見せる。Blomfieldの原文キーワードをそのまま体現できる

**案B：Self-updating（陳腐化しないBrain）**
- Company Brainの最大の弱点は「すぐ古くなる」こと。ここを解く
- Slackで新しい決定が下されると、Brainが「既存のPlaybookと矛盾してますよ、更新しますか？」と提案 → 承認で知識が自己更新
- デモ：古いルール → チャットで新方針が出る → Brainが差分検知して更新提案
- **勝ちポイント**：「keeps it current」というYCの要求に真正面から答える。技術的に地味に難しいので "Engineering Lift" 賞も狙える

**案C：オンボーディング特化の縦切り**
- 全社は広すぎるので「新入社員の最初の1週間」に絞る
- 新人の質問にBrainが答え、しかも「誰に聞けばいいか」「関連する過去の議論」まで提示
- **勝ちポイント**：ペルソナが明確でデモが刺さりやすい。5時間で完成度を出しやすい

### 落とし穴
- 「Notionと繋いでQ&A」だけだと**ただのRAG**になり、審査員に「で、Gleanと何が違うの？」と刺される。必ず **executable / self-updating** のどちらかで差別化を。

## テーマ2. Dynamic Software Interfaces（Ankit Gupta）

### YCが本当に言っていること

> 従来のソフトは "one-size-fits-all"（万人に同じUI）。Netflixはコンテンツはパーソナライズするが、UIレイアウトは全員同じ。今までUIをカスタマイズできるのは、フォワードデプロイエンジニアがつく**エンタープライズだけ**だった。

そして核心のビジョン：

> **コーディングエージェントが十分賢くなった今、ユーザー全員が "自分専用のフォワードデプロイエンジニア" を持てる。** ユーザーは、同じ土台（primitives）を共有しながら、自分のユースケース向けに**まったく違うインターフェース**を作れる。

YCが欲しいのは「ソフトウェア配信のスタック全体を作り直す radical thinker」。つまり **UIが固定物ではなく、ユーザー/エージェントがその場で生成・改変する動的なもの**になる世界。

### なぜ今か
- コーディングエージェントがUIを即席生成できるようになった（v0, Claude Artifactsの発想）
- 「ソフトを使う」＝「開発者が決めた画面を使う」から「自分に最適化した画面を生成する」へ
- 元の会話にあった「同じメールアプリが、人によってタスク管理UIにもカレンダーUIにも見える」がまさにこれ

### プロダクト案（5時間で作れる粒度）

**案A：「自然言語でUIを作り替えられるアプリ」（本命・UX賞狙い）**
- 1つのデータソース（例：タスク、メール、CRMの顧客リスト）に対して、ユーザーが「これをカンバンで見せて」「営業向けにこの3列だけのダッシュボードにして」と言うと、**その場でUIが再生成される**
- 裏側：データは共通スキーマ（primitives）、UIだけをLLMが生成してレンダリング
- デモの魔法：同じデータが、話しかけるたびに全く違うアプリに変身する瞬間
- **勝ちポイント**：デモ映えが圧倒的。90秒動画で「うわ」と言わせられる。UX/UI賞ど真ん中

**案B：ユーザーごとに"育つ"UI（適応型）**
- ユーザーの操作ログをエージェントが観察 →「あなたはいつもこの3操作をしてますね、ボタンにまとめました」とUIが自己最適化
- 明示的な指示すら要らず、使うほどUIが自分に寄っていく
- **勝ちポイント**：「radical thinker」感が強い。投資対象としてのビジョンが大きい

**案C：エージェント向けの動的UIレイヤー（Gupta原文の隠れた本命）**
- 原文には *"software that can be accessed by the user's coding agents"* という一文がある。つまり**人間だけでなくエージェントもUIをカスタマイズする**
- 案：あるアプリに対し、ユーザーのエージェントが「この人はこう使うから、こういうUIを組んどこう」と**エージェントがUIを生成・設定する**基盤
- **勝ちポイント**：テーマ2とテーマ3の交差点。Sean Grove（エージェント制御）に刺さりやすい

### 落とし穴
- 「テーマ生成器」「ダッシュボードビルダー」に見えると弱い。ポイントは **同じprimitivesの上で根本的に違う体験が立ち上がる**こと。「レイアウトを変える」ではなく「別のアプリになる」レベルの差を見せる。

## テーマ3. Software for Agents（Aaron Epstein）

### YCが本当に言っていること

> 今のAIエージェントは「人間がボタンをクリックする前提で作られたソフト」を操作している。だから**遅く、不安定で、壊れやすい（slow, inconsistent, brittle）**。エージェントに必要なのは視覚的ダッシュボードではなく、**API・MCP・CLIといった機械可読インターフェース**。

そして最大のメッセージ：

> **人々が今日使っているソフトの主要カテゴリは、すべてエージェント向けに作り直される必要がある。** 一番大きなチャンスは「エージェント自体」を作ることではなく、**そのエージェントが依存するソフトを作ること**。
> 「Making Something Agents Want（エージェントが欲しがるものを作れ）」

YCの有名なスローガン "Make Something People Want" の**エージェント版**をわざわざ掲げているのがポイント。ユーザーが人間じゃなく**エージェント**になる、という世界観です。

### なぜ今か
- 「AI wrapper」の次の波として「AIのためのインフラ／SaaS」に投資マネーが向かっている
- 既存SaaSはブラウザ自動化で無理やり操作されていて非効率 → ネイティブにエージェント対応した方が速く確実
- GStack/Codex MCP の流れがまさにこの感覚

### プロダクト案（5時間で作れる粒度）

**案A：「人間用SaaSを1コマンドでエージェント対応にする」ラッパー基盤（本命）**
- 既存のWebアプリやAPIを食わせると、**自動でMCPサーバー/CLIを生成**する
- エージェントが「認証 → 実行 → 結果検証」までプログラム的にできるようになる
- デモ：適当なWeb APIを1つ渡す → 数十秒でMCPツールが生えて、Claude/エージェントがそれを使ってタスクを完遂
- **勝ちポイント**：Aaron原文の「エージェントが依存するソフトを作れ」に完全一致。devtools審査員（Sean）に強い

**案B：エージェント専用の「◯◯」を一個作る（縦切り・分かりやすい）**
- 既存の人間用サービスを1つ選び、**エージェントファーストで作り直す**
  - 例：エージェント用の決済（人間のカード入力UIなし、APIとポリシーだけ）
  - 例：エージェント用のブラウザ/スクレイピング（人間用の画面を返さず構造化データを返す）
  - 例：エージェント用のサインアップ（CAPTCHAやフォームでなく、認証トークン発行API）
- **勝ちポイント**：「人間用UIを剥ぎ取ったらこうなる」という具体例が刺さる。何を作るか一目で分かる

**案C：エージェント向けの「信頼と検証」レイヤー（Sean Grove ど真ん中）**
- エージェントがソフトを操作した結果が**本当に正しいかを検証**する仕組み（人間用ソフトは目視でOKだが、エージェントには機械可読な検証が要る）
- 例：エージェントのアクションログを受け取り、「この操作は仕様通りか」を自動テスト・ロールバックできるサンドボックス
- **勝ちポイント**：原文の "brittle（壊れやすい）" を解く。エンジニアリングの重さで勝負できる

### 落とし穴
- 「MCPサーバーを1個書きました」で止まると弱い。**「あらゆるソフトがこう作り直される」というプラットフォーム的な広がり**を、たとえ1例でも示すこと。

## どれを選ぶべきか（おすすめ）

5時間・審査員・デモ動画90秒という制約から、こう整理できます。

| テーマ | デモ映え | 5hでの作りやすさ | 差別化の難しさ | 狙える賞 |
|---|---|---|---|---|
| **1. Company Brain** | 中 | 中（データ準備が要る） | 高（RAGと差別化が難所） | Investable / Eng Lift |
| **2. Dynamic Interfaces** | **高** | 高（フロント中心） | 中 | **UX/UI** |
| **3. Software for Agents** | 中〜高 | 中 | 中 | **Eng Lift / Investable** |

**個人的なおすすめは、チーム構成で決める**のが一番です。

- **フロント/デザインが強いチーム → テーマ2（Dynamic Interfaces）案A。** 「話しかけるとアプリが変身する」デモは90秒動画で圧勝しやすく、UX/UI賞が現実的。
- **バックエンド/インフラが強いチーム → テーマ3（Software for Agents）案A。** 審査員（Sean・Henry）と最も相性がよく、「投資対象になる」ストーリーを描きやすい。YCの原文キーワードにも一番忠実。
- **課題ドリブンで攻めたいチーム → テーマ1（Company Brain）案B（self-updating）。** 技術的に難しく差別化しやすいが、データ準備で時間を溶かすリスクあり。

もし「YCへの本気応募」まで見据えるなら、**テーマ3が一番"会社の種"に見えやすい**（市場が明確で、投資家が今まさにお金を入れている領域だから）。一方、**このハッカソンで勝つ確率を最大化するならテーマ2**（限られた時間でインパクトを出しやすい）というのが私の見立てです。
